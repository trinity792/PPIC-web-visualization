## What the Module Produces

The pipeline builds one file: `data/data-cleaned/building-permits/BuildingPermits_Current.csv`. It records **residential building permits** - monthly counts of authorized new housing units, split by structure size (1 unit, 2 units, 3-4 units, 5-or-more, and their total) - for California metropolitan areas and for all 50 US states. The generated contract holds **197 months (2010-01 through 2026-05), 14,691 rows**.

This is the simplest of the five modules on the *measurement* axis (no race, tenure, or age dimension - just five structure-size counts) and the most distinctive on two others:

- **It is the only monthly module.** The `Date` column is `YYYY-MM`, not a year.
- **Its deep history cannot be rebuilt from the live source.** The Census only hosts a rolling ~2-year window of monthly `.xls` files (as of 2026-07, back to 2024-01; earlier months return 404). So 2010-01 through 2023-12 lives in a dedicated, read-only artifact - `BuildingPermits_Historical.csv` - seeded once from the legacy accumulated snapshot; the live pipeline owns 2024-01 onward, and Phase 1 composes the two each run (preferring live rows). Because that seed is a distinct file the pipeline never writes, the deep history survives even a lost or truncated `Current.csv`, and a Phase 5 guard fails the run loudly if any seeded month ever goes missing from the output. See "From Initial Refactor to Current State."

The refactor's defining win was structural: the legacy module copy-pasted the full acquire → clean → merge → derive → save sequence **six times** (the try and except branches of all three `visualize_*` functions), with the two cleaners existing in four copies. That collapses here into one orchestrator with individually testable phases, fallback handled once in the acquisition layer, and saving decoupled entirely from charting.

## Data Sources

One external source feeds the module - the **U.S. Census Bureau Building Permits Survey (BPS)** monthly release, fetched as two `.xls` files per month:

| File | URL pattern | Content | Kept |
|---|---|---|---|
| CBSA monthly | `.../bps/xls/cbsamonthly_{YYYYMM}.xls` | Permits by Core-Based Statistical Area | California **metropolitan** CBSAs (micropolitan, `Metro /Micro Code == 5`, dropped) |
| State monthly | `.../bps/xls/statemonthly_{YYYYMM}.xls` | Permits by state | The 50 states |

BPS publishes monthly with a ~2-month lag. Parsing the legacy `.xls` format requires `xlrd>=2.0.1` in the pipeline environment.

**Availability caveat:** the monthly endpoints host only a rolling ~2-year window. The pipeline acquires only the months **between the last stored month and the latest available month**, and it **skips** (with a log) any month that returns a 404 rather than aborting - so the 2010-2023 months, which now 404, are simply not re-fetched and are supplied by the seed instead.

## The Dataset: Grain, Columns, and Geography

**Grain:** one row per `(Date, Geographic Level, Location)`.

**Columns (in output order):**

| Column | Meaning |
|---|---|
| `Geographic Level` | `State` or `Metro` |
| `Location` | A state name (`California`, `Texas`, ..., 50 states) for `State` rows, or a canonical CA metro display name (`Los Angeles`, `San Francisco`, `Inland Empire`, `Sacramento`, `Bakersfield`, ...) for `Metro` rows |
| `Date` | Month as `YYYY-MM` |
| `Total`, `1 Unit`, `2 Units`, `3 and 4 Units`, `5 Units or More` | Integer counts of authorized housing units. `Total` is the source total, not a recomputed sum |

**Derived and aggregated values are NOT stored.** `2+ Units`, `Rest of US`, index-to-100, the trailing-12-month ("year-to-date") rolling sum, two-period change, **and the 9-region aggregate** are all computed downstream in the frontend data-access layer, never persisted. This keeps the stored dataset at its minimal native grain.

**Geography:**

- **State:** 50 states, sourced directly from the state monthly file.
- **Metro:** CA metropolitan CBSAs at native BPS grain, using canonical display names from the shared `california_geography.cbsa_metros`. The grain is "up to 26" - **Madera** was de-delineated as a standalone MSA and no longer appears in current data, though older seeded months still carry it, so validation allows a *subset* of the canonical 26.
- **Region:** a **frontend aggregate, not stored** - the data-access layer sums member metros into the 9 shared regions on demand. Region aggregates cover metropolitan counties only and **under-count** rural counties covered by no CBSA (a documented, surfaced caveat).
- **County:** not stored and not derivable (a multi-county CBSA total cannot be split into its counties).

**Metro name drift is absorbed by code-based renames.** The SF metro is now published as "San Francisco-Oakland-Fremont" (was "-Berkeley"), Bakersfield as "Bakersfield-Delano", Stockton as "Stockton-Lodi". A rename map keyed on the stable **CBSA code** (12540, 41860, 44700) pins these to canonical display names regardless of the Census label churn.

## How the Pipeline Runs: Five Phases

The orchestrator `build_building_permits_dataset()` runs five phases. Each is wrapped so any exception re-raises as a `BuildingPermitsPipelinePhaseError` tagged with the phase name.

### Phase 1 - Setup and Load

Resolve the three config dicts (`paths`, `sources`, `schemas`) and the shared California geography, read the live output (`BuildingPermits_Current.csv`) and the immutable deep-history seed (`BuildingPermits_Historical.csv`), **compose** them into one `historical` frame (preferring live rows on any overlap, while the seed supplies any deep month the live file has lost), and find the latest stored month. If neither file exists, a loud warning notes the run is proceeding on live data alone.

### Phase 2 - Acquisition

Resolve the latest published BPS month by probing backward from the current calendar month (a 404 advances the probe; a **transient network fault is retried, then fails loud**; a parse error raises). The probe hands back the files it downloaded so the resolved month is not re-fetched. Enumerate the months strictly after the last stored month through the latest available one, then download each month's CBSA and state files. On a cold start (empty history) the enumeration reaches back to `earliest_month`, but any not-yet-hosted month is skipped (routed through the logger); if a live download fails outright, acquisition falls back to the last-saved rows and flags `source_failed`.

### Phase 3 - Clean and Tag

Clean each monthly CBSA frame (reseat the header, pin the CBSA/micro code columns to a stable integer dtype, split `Name` into location/state, keep CA metros, coerce the five measures to int with a located error on a bad cell, apply the code-based and display renames, stamp `Date`) and each state frame (select and rename the six columns, filter to the 50 states, coerce measures, stamp `Date`), run the **per-frame cleaning validator** on each (Date format, null keys, integer non-negative measures, known locations), validate every metro name against the shared config, and concatenate into one `Geographic Level`-tagged frame.

### Phase 4 - Merge

Atomically replace any overlapping stored month (drop every historical row whose `Date` is in the incoming frame, append the incoming rows whole - never a key-level upsert).

### Phase 5 - Finalize and Save

Enforce contract column order and integer types, detect whether the prepared output differs from the saved current file (using the **same serialized-CSV comparison the write step uses**), assert that every seeded deep-history month survived into the output (fail loud otherwise), run final validation (row-count bounds, both levels present, 50 states per month, metros within the canonical set, a **contiguous** monthly range across the present series, non-negative measures, no duplicate keys), and - only if new data was detected - archive the prior file with a timestamp and write the new CSV **atomically** (temp file, then rename).

## Key Calculations

There is very little math in the stored dataset by design - the five counts come straight from the BPS spreadsheet and `Total` is the source's own total. The interesting logic is elsewhere:

- **Header reseating and named selection.** The BPS spreadsheet has a banner above the real header, and it repeats each measure name across a "Current Month" block and a "Year to Date" block. The metro cleaner locates the true header row by finding the row carrying `Name` and `CBSA`, keeps only the first (current-month) occurrence of each duplicated column, and then selects by **name** - raising if an expected column is absent. This replaces the legacy positional slicing (`df.iloc[6]`, `df.iloc[:, 0:8]`) that broke silently on a layout change.
- **Forward month enumeration.** `months_to_acquire` enumerates months *forward* from the last stored month, so no month is skipped or re-downloaded - replacing the legacy "decrement from now until you hit the stored month" walk with its bare `except`.
- **Metro to region aggregation** (frontend only) sums member metros using the shared `metro_to_region_mapping`, which is derived by composing `metro_to_county_mapping` with the existing `regions_mapping` (every CA CBSA is a union of whole counties nesting within one region).

## Architecture and Boundaries

The module follows the standard three-layer V3 shape - shared -> `building_permits` -> orchestrator - and this migration **extended the shared layer**: the CBSA-metro reference data (`cbsa_metros`, `metro_to_county_mapping`, `metro_to_region_mapping`) was lifted out of the legacy module into `scripts/shared/geography/california_geography.py` so the metro grain is owned centrally, with a JS mirror (`lib/geography/californiaGeography.js`) so the Python and frontend groupings never drift.

There is **no Python visualization layer** - the legacy `visualize_line/bar/map()` methods (each of which also triggered the save as a side effect and carried a full duplicate pipeline in its `except` branch) were dropped. The frontend triad (`lib/visualization/moduleSchemas/buildingPermits.js`, `lib/data/building_permits.js`, `app/api/building-permits/route.js`) only ever *reads* the contract CSV and owns all the month-aware shaping and derived values.

## Phase 2: Acquisition

### `scripts/building_permits/acquisition/census_bps_downloader.py`

- **`BPSMonthUnavailableError`** - signals "not published" (404 / missing file), distinct from a parse failure.
- **`_is_missing_file_error(error)`** - branches on the `HTTPDownloadError`'s structured `status_code == 404` (the shared downloader carries it), falling back to a message match only when no status code is present. This distinguishes "not published" from a transient fault by the actual HTTP status rather than by sniffing text.
- **`_download_month(url_pattern_key, year, month, ...)`** - formats the `{yyyymm}` URL, fetches, translates a 404 into `BPSMonthUnavailableError`, and parses with `pd.read_excel` (a malformed-but-present file raises `ValueError`, never masqueraded as "not published").
- **`download_cbsa_month` / `download_state_month`** - thin wrappers over `_download_month`. *Derived from* the download step inside the two legacy `clean_and_scrape_*` walks.

### `scripts/building_permits/acquisition/source_fallback.py`

- **Month arithmetic:** `_month_index`, `_from_month_index`, `_previous_month`.
- **`_probe_month(...)`** - downloads one month's CBSA and state files, **retrying a transient `HTTPDownloadError` up to `retry_attempts` times before re-raising**. A `BPSMonthUnavailableError` (404) propagates immediately so the caller advances the probe; a parse `ValueError` is not caught, so a malformed release fails loud without being retried.
- **`resolve_latest_month(source_settings, headers, timeout, max_month_lookback, probe_retry_attempts=2, logger=None)`** - probes backward from the current month, accepting the first month whose CBSA **and** state files both download and parse; only `BPSMonthUnavailableError` advances the probe. Returns `(year, month, prefetched)`, where `prefetched` maps the resolved `"YYYY-MM"` to its already-downloaded `(cbsa_df, state_df)` so `acquire_months` need not re-fetch it. Raises if nothing is found in the window. *Derived from* the legacy `datetime.now()` bare-`except` decrement.
- **`months_to_acquire(last_stored_month, latest_available_month, excluded_months=None)`** - forward-enumerates the months strictly after the last stored one through the latest available. *Replaces* the legacy backward walk.
- **`acquire_months(months, download_cbsa_fn, download_state_fn, saved_rows_fn, prefetched=None, logger=None)`** - downloads each month's two files, reusing any month present in `prefetched`; **skips** a `BPSMonthUnavailableError` month (routing the notice through the logger); on any other exception falls back to the last-saved rows and returns `source_failed=True`. Returns `(cbsa_frames, state_frames, source_failed)`. *Derived from* the legacy scrape loop and its ad-hoc fallback.

## From Initial Refactor to Current State

The refactor first collapsed the legacy `permits_code.py` - which copy-pasted the full acquire → clean → merge → derive → save sequence six times - into the phase modules described above. A later reliability, robustness, and efficiency pass then hardened several parts of that initial refactored state into what this guide documents. This section records those changes: what the first-cut refactor left in place, and what it became, so a reader understands why the current code is shaped the way it is. Two of the changes below reflect explicit maintainer decisions, noted inline.

### Deep history became a separate, seeded, atomically-written artifact

The first-cut refactor pointed `historical_data_path` at the pipeline's own output (both were `BuildingPermits_Current.csv`), so each run read the contract as its own baseline and wrote back to it. Combined with the source's ~2-year rolling window - the `cbsamonthly`/`statemonthly` endpoints 404 for every pre-2024 month, so the deep history genuinely cannot be re-fetched - this was the sharpest data-integrity risk of the five modules: **losing or truncating `Current.csv` meant losing 2010-2023 permanently**, and a cold start (enumerating back to `earliest_month`, then skipping every 404) would have produced only the recent rolling window while final validation's contiguity check - which spans only the *present* range - accepted the truncated series silently.

Per the maintainer's decision (option **a**: commit the seed as a permanent read-only artifact and rebuild the baseline as seed+live each run), the pre-2024 deep history now lives in its own file, `BuildingPermits_Historical.csv` (`historical_data_path`, decoupled from `current_data_path`), which the pipeline **never writes**. `load_historical_baseline` reads it, and Phase 1's `compose_baseline` unions it with the live output, preferring live rows on any overlapping key while the seed supplies any deep month the live file may have lost - so the deep history is recoverable from the committed seed alone even if `Current.csv` is destroyed. Three guards backstop this:

- **The write is now atomic.** `archive_and_save` stages the new CSV into a `.tmp` sibling and `Path.replace`s it into position (archive old → write tmp → atomic rename), so a crash mid-write can never leave a truncated `Current.csv`. Of all the non-atomic-write findings across the five modules this was the one to fix first, because it is the only one where a single bad write destroys data that cannot be regenerated. A regression test simulates a failed replace and asserts the original file is intact.
- **A loud deep-history guard.** Phase 5 asserts every month present in the seed survives into the prepared output; if any seeded month is missing it raises rather than shipping a truncated series - catching the wholesale loss the contiguity check cannot see.
- **A cold start warns instead of silently under-producing.** If neither the live output nor the seed exists, Phase 1 emits a loud `UserWarning` and proceeds on live data alone, so a fresh checkout is explicit about yielding only the rolling window.

### The pre-clean validator became a wired acceptance gate

`validate_cleaning_output` was defined and tested but had no caller - the orchestrator imported only the final validator, so its per-frame `YYYY-MM` Date-format, null-key, non-negative-integer, and known-location checks never ran on live data. It is now **wired into Phase 3**, run on both the cleaned metro and state frames before the merge. This closes a real gap: the final validator checks only non-negativity on the incoming rows, not their date format or integer-ness, and it catches an unknown location before `validate_metro_names` (which raises rather than returning messages) is reached.

### The cleaner casts became located and dtype-stable

Two fragilities lived in the measure casts. First, both cleaners cast the five measures with a bare `.astype(int)`, so a single blank or non-numeric cell raised an opaque `ValueError: invalid literal for int()` with no hint of which row failed. The cast now runs through the shared `coerce_measures_to_int`, which `pd.to_numeric(errors="coerce")`s each column and raises a **located** error naming the `(Date, Location, column, value)` - the strictness is unchanged (a bad cell still fails the run), but a partially-populated BPS row is now diagnosable at a glance.

Second, the metro cleaner's code-based rename (`_CBSA_CODE_RENAMES`, int keys `12540`/`41860`/`44700`) and micropolitan filter (`!= 5`) depended on the `CBSA`/`Metro /Micro Code` cells being read as ints - but `xlrd` commonly reads numeric `.xls` cells as floats (`41860.0`, `5.0`), in which case the int-keyed lookup would silently miss and the rename would no-op, leaving an unrenamed SF label that then fails `validate_metro_names`. The cleaner now pins both code columns to a nullable-integer dtype (`pd.to_numeric(...).astype("Int64")`) before the filter and the rename (applied via a dtype-stable `.map`), so the behavior no longer rides on the parser's incidental numeric type. A test feeds float-typed codes and asserts the renames still fire and the micropolitan row is still dropped.

### The change flag shares the write gate's definition of "changed"

`detect_new_data` sorted both frames and used `DataFrame.equals`, comparing freshly-built in-memory int measures against dtypes inferred from `pd.read_csv` - a drift that could report spurious "new data" on an unchanged dataset (harmless, since `archive_and_save` independently no-ops on identical text, but it made the flag untrustworthy and always triggered the archive-compare work). It now compares the **serialized CSV text** - exactly what the write step compares - after aligning columns and sorting to the canonical grain, so the change flag can never disagree with whether a byte-different file is actually written. The offline idempotency check confirms it: composing and preparing the live 14,691-row dataset reports no change and is byte-identical to the file on disk.

### The latest-month probe gained retry, single-fetch, and status-code classification

Three acquisition-seam items were addressed together. The probe (`resolve_latest_month`) advanced only on `BPSMonthUnavailableError`, so a transient non-404 fault - a timeout or connection blip on the newest month - propagated and aborted Phase 2 outright, even though `acquire_months` degrades on a generic error. Per the maintainer's decision (**fail loud with retry**), `_probe_month` now retries a transient `HTTPDownloadError` `probe_retry_attempts` times (default 2) before re-raising, so a single blip recovers while a genuine outage still fails the run rather than silently capping the series at a stale month (a parse `ValueError` is still not retried - a malformed release fails loud immediately). The probe also downloaded and discarded the files it fetched, which `acquire_months` then re-downloaded; `resolve_latest_month` now returns those frames as a `prefetched` map so the resolved month is fetched once. And `_is_missing_file_error` now branches on the shared `HTTPDownloadError`'s structured `status_code == 404` rather than substring-matching the message, so 404-vs-transient classification no longer rides on the downloader's message format.

### The skipped-month notice reached the logger, and the clean-degrade path is locked by a test

`acquire_months` reported skipped (unpublished) months via `print` - the normal cold-start path, which can be hundreds of 404s. It now routes the notice through `log_message` (accepting a `logger` argument), so it lands in the structured run record instead of stdout. (The rest of the logging layer was already fully wired before this pass - Phase 2 `log_processing_step`, Phase 5 `log_data_quality_check`, and `execute_pipeline_run` appending a JSONL record with `source_failed`; this orchestrator remains the best-instrumented of the five.) Finally, the module's one genuine strength - the fallback degrades cleanly rather than crashing (a non-dict saved payload flows through `_clean_monthly_frames`' guard into empty frames, an unchanged merge, and no write) - is now protected by an end-to-end regression test: a forced live-download failure finishes on the existing history with `source_failed=True` and no write, and a companion test proves the seed alone rebuilds the deep history when `Current.csv` is missing.
