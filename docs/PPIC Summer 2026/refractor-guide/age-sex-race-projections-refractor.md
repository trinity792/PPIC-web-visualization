## What the Module Produces

The pipeline builds one file: `data/data-cleaned/demographic-projections/DemographicProjections_Current.csv`. It holds population counts broken out by **location, year, age group, sex, and race/ethnicity**, drawn from two sources and stacked into a single long-format table. A full run writes on the order of 1.5 million rows (roughly 1.58 million when both sources succeed).

The dataset blends two fundamentally different kinds of number:

- **Projections** - the California Department of Finance (DoF) P-3 series, which projects the state's population forward from 2020 to 2070.
- **Estimates** - the U.S. Census Bureau cc-est series, which records estimated population for all 50 states for the recent past (2020-2025).

Because the two are conceptually different (one looks forward, one looks back), every row carries a `Source` label so the frontend can keep them visually distinct and never let a user mistake a projection for observed data.

The refactor dissolved the legacy `Projections` class - whose entire ETL ran inside `__init__()` and whose `visualize_*` methods drew charts directly - into a set of single-responsibility functions coordinated by an orchestrator. Charting moved out of Python entirely and into the shared React frontend.

## Data Sources

| Source | Format | Geography | Years | Grain in the raw file |
|---|---|---|---|---|
| DoF P-3 projections | Zip archive containing one flat, comma-delimited CSV | 58 California counties (FIPS 6001-6115) | 2020-2070 (Baseline 2024, Vintage 2026) | One row per county, single-year age (0-110), sex, and 7-code race |
| Census cc-est estimates | Official wide-format `CC-EST{VINTAGE}-ALLDATA.csv`, Latin-1 encoded | All U.S. counties (aggregated to 50 states) | 2020-2025 | One row per county, coded year, coded 5-year age group, with race-by-sex population in wide columns |

The P-3 file is by far the larger input. Its raw dimensions are 58 counties x 51 years x 2 sexes x 7 race codes x 111 single-year ages, which is roughly 4.6 million rows before the pipeline bins single-year ages down to 18 five-year groups.

Each source is acquired independently through a **three-step fallback cascade**: try the live download strategies in order, then a manually placed CSV, then the last-saved rows already in the contract file. A source that falls all the way back to last-saved is flagged as "failed" so the run can still finish (with stale data for that source) rather than crash.

## The Dataset: Grain, Columns, and Geography

**Grain:** one row per `(Geographic Level, Location, Year, Age Group, Sex, Race/Ethnicity, Source)`.

**Columns (in output order):**

| Column | Meaning |
|---|---|
| `Geographic Level` | `County`, `Region` (9 CA regions), `State` (California from DoF), or `US State` (the 50 states from Census) |
| `Location` | County name, region name, `California`, or a U.S. state name |
| `Year` | Integer |
| `Age Group` | One of 18 canonical 5-year labels (`0-4` ... `80-84`, `85+`) or the aggregate `All Ages` |
| `Sex` | `Male`, `Female`, or the aggregate `Both Sexes` |
| `Race/Ethnicity` | `White`, `Black`, `Asian`, `NHPI`, `AIAN`, `Multiracial`, `Hispanic`, or the aggregate `All` |
| `Population` | Integer count |
| `Source` | `DoF P-3` or `Census cc-est` |

**Why California appears twice.** The DoF rollup of the 58 counties produces a `California` row at `Geographic Level = State`. The Census 50-state series also contains California, but at `Geographic Level = US State`. Both are legitimate and are distinguished by `Source`, so geographic-level assignment must look at **both** `Location` and `Source`.

**Aggregate ("marginal") rows are pre-computed and stored.** Rather than force the frontend to sum on every request, the pipeline writes the `All Ages`, `Both Sexes`, and `All`-race totals directly into the CSV. Coarser age presets (Under 18, 18-25, 26-64, 65+) are *not* stored - the API sums the stored 5-year groups into those presets on request.

**Age storage is lossy by design.** Single-year P-3 ages are binned to 5-year groups during cleaning (summing population within each bin) before anything is written. The `85+` bin folds together ages 85 through 110.

## How the Pipeline Runs: Five Phases

The orchestrator `build_projections_dataset()` runs five phases in sequence. Each is wrapped so that any exception is re-raised as a `ProjectionsPipelinePhaseError` tagged with the phase name, which makes a failure immediately traceable to a stage.

### Phase 1 - Setup and Load

Resolve the three config dicts (`paths`, `sources`, `schemas`), then build the `historical` DataFrame by **unioning the immutable seed (`DemographicProjections_Historical.csv`, if present) with the current output** — preferring current rows on any overlap while the seed supplies any deep year the current file may lack. That single frame is reused everywhere a fallback might need last-saved rows, so the large file is parsed only once per run. With neither file present the pipeline cold-starts on live data alone.

### Phase 2 - Acquisition

For each source, run the fallback coordinator. The **live strategies** differ by mode:

- **Online (default):** DoF tries link-text discovery, then a positional HTML fallback; Census constructs the canonical ALLDATA URL from the newest vintage directory. Each downloads (or reuses a cached file) and returns a path.
- **Offline** (`config={"offline": True}` or `PROJECTIONS_OFFLINE=1`): DoF reuses a local extracted P-3 CSV or extracts a local zip; Census reuses a local cc-est CSV. This lets the whole pipeline run from disk with no network.

If every live strategy fails, the coordinator tries a manually placed CSV, then last-saved rows. Live and manual paths hand a **file path** to the next phase; only the last-saved fallback hands back an already-cleaned **DataFrame** (paired with a `source_failed=True` flag so cleaning is skipped for it).

### Phase 3 - Cleaning

Each source is cleaned to the same canonical schema:

- **DoF P-3:** validate the header and values, map FIPS codes to county names, decode the 7 race codes, standardize sex labels, and bin single-year ages into 5-year groups (summing population). Output is tagged `Geographic Level = County`.
- **Census cc-est:** parse the Latin-1 wide file, keep only `SUMLEV=050` county rows, sum the 14 race-by-sex population columns to state totals by `(STNAME, YEAR, AGEGRP)`, confirm all 50 states are present, then reshape wide-to-long and decode the year and age-group codes. Output is tagged `Geographic Level = US State`.

If a cleaner raises, Phase 3 retries with the manual file, then falls back to last-saved rows (reduced back to base strata so aggregation doesn't double-count).

### Phase 4 - Merge and Aggregate

1. **Completeness gate + atomic merge.** Each cleaned source is validated for stratification completeness (every location/year must carry the full 18 x 2 x 7 base matrix), then merged with historical rows one full year at a time - overlapping years are replaced wholesale, never patched key by key.
2. **Concatenate** the DoF and Census frames.
3. **Geographic rollups** on the county rows: `add_state_total` sums the 58 counties into a `California` `State` row, and `add_regional_data` sums counties into the 9 CA `Region` rows. Both run unconditionally.
4. **Precomputed marginals:** `build_precomputed_totals` appends `All Ages`, then `Both Sexes`, then `All`-race rows, in that order so each aggregate correctly includes the previous ones.
5. **Change detection** reduces both the freshly cleaned rows and history to the same base strata, then compares them to set the `dof_new_data` / `census_new_data` flags. Because both sides are compared like-for-like, the flags are meaningful — an unchanged source reports `False`, and Phase 5 then skips the write entirely.

### Phase 5 - Finalize and Save

Assign each row's `Geographic Level` from `Location` + `Source`, enforce the contract column order and types, and run final validation (required columns, row-count bounds, expected geographic levels, year range against the soft sanity bounds, no duplicate keys). The set of **expected levels is derived from which sources actually succeeded**, so an offline DoF-only run still validates (it doesn't demand `US State`). If (and only if) new data was detected, `archive_and_save` timestamps the prior file into the archive directory and writes the new CSV **atomically** — but only if the bytes actually differ.

## Key Calculations

- **Age binning (lossy).** `pd.cut` with edges `[0, 5, ..., 85, inf]`, right-open, assigns each single-year age to a 5-year label; the pipeline then groups and sums population within each bin. Ages 85-110 collapse into `85+`.
- **County to region to state rollups.** Region and California-state populations are always summed from county rows (the P-3 source has no region or state row of its own). Census state rows are built earlier, from county observations, inside the Census cleaner.
- **Precomputed marginals cascade.** `All Ages` is summed first; `Both Sexes` is then summed over a frame that already contains `All Ages`; `All`-race is summed over a frame that already contains `Both Sexes`. This ordering is what makes the "grand total" rows (for example, `All Ages` + `Both Sexes` + `All`) internally consistent.
- **Stratification completeness.** For each `(Geographic Level, Location, Year, Source)` group, the validator counts distinct base `(Age Group, Sex, Race/Ethnicity)` tuples and compares against `18 x 2 x 7 = 252`. Aggregate rows are excluded from the count so precomputed totals can never mask a missing base stratum.

## Architecture and Boundaries

The module follows the standard three-layer V3 shape:

- **Shared layer** (`scripts/shared/`) - dataset-agnostic helpers: HTTP downloads, California geography, generic DataFrame validators. These never import anything projection-specific.
- **Domain layer** (`scripts/projections/`) - config, acquisition, cleaning, aggregation, merging, validation, and output modules specific to this dataset.
- **Orchestrator** (`scripts/orchestrators/projections_pipeline.py`) - wires the domain modules into the five phases and owns the fallback and offline logic.

The dependency direction is strict: shared -> projections -> orchestrator. Unlike the legacy module, there is **no Python visualization layer** - the legacy `visualize_line/bar/map()` methods were dropped and replaced by the shared React charting UI. The frontend deliverables (`lib/visualization/moduleSchemas/demographicProjections.js`, `lib/data/demographic_projections.js`, `app/api/projections/route.js`) consume the contract CSV over an HTTP API and are described at the interface level in the second half.

## Configuration Modules

### `scripts/projections/config/paths.py`

**`get_paths()`** - builds every pipeline path from `lib.config.get_project_paths()`: the cleaned/raw/archive directories under `demographic-projections/`, the contract path `DemographicProjections_Current.csv`, and the two manual-fallback paths (`P-3_Downloaded.csv`, `cc-est_Downloaded.csv`). Returns `pathlib.Path` objects.

> Note: `historical_data_path` now points at a **separate** immutable seed (`DemographicProjections_Historical.csv`), decoupled from `current_data_path`. Phase 1 unions the seed with the current output (preferring current), so the pipeline no longer reads *only* its own prior output as history. The seed is optional — its absence cold-starts cleanly on live data. See *From Initial Refactor to Current State → A separate, seeded history path*.

*Legacy lineage:* replaces the hard-coded `R:\UCF\...` Windows paths scattered through the legacy `__init__`.

### `scripts/projections/config/sources.py`

**`get_source_settings()`** - returns the DoF and Census base URLs, HTTP headers/timeout (from `lib.config.get_default_http_settings()`), cache windows (P-3 90 days, cc-est 30 days), a longer `ccest_download_timeout` (>= 300s, because the national ALLDATA file is ~100 MB from a slow host), the P-3 filename pattern, the expected raw-column lists (imported from `schemas.py`), and the change-detection boundary years (both 2019, so every observed year from 2020 on stays in scope).

*Legacy lineage:* consolidates the scattered URL literals and `sheet_names[1]` assumptions from `scrape_dof_projections()` / `scrape_census_estimates()`.

### `scripts/projections/config/schemas.py`

Owns all reference constants and the one accessor:

- Module constants: `P3_RAW_COLUMNS`, `CCEST_RAW_COLUMNS` (7 identifiers + 14 race-by-sex fields), the 58 FIPS-ordered county names, the 50 state names, canonical age groups, age-bin edges, canonical race groups and sexes, the P-3 race7 map, the Census race-prefix map, and the Census year-code map.
- **`_build_fips_to_county_map()`** - zips `range(6001, 6116, 2)` with the alphabetical county list (California FIPS are assigned alphabetically, so index N -> FIPS 6001 + 2N).
- **`get_schema_config()`** - assembles everything above plus the cleaning- and final-validation config blocks into one dict consumed by cleaning, aggregation, validation, and output. The Census year is now **derived arithmetically** (`census_year_base = 2018`, so calendar = `2018 + code`) with a `census_year_sane_range` guard rather than a capped `{code: 2018 + code for code in range(2, 8)}` table, so a new vintage (code 8 → 2026) ingests without a config edit; the age-group code map is `enumerate(canonical_age_groups, start=1)`. The P-3 horizon is bounded by `p3_year_sane_bounds` + `p3_max_year_span` rather than a hard `(2020, 2070)`. A module-import parity assertion ties `_AGE_BIN_EDGES` to `_CANONICAL_AGE_GROUPS` so the two age-label routes can never drift.

*Legacy lineage:* replaces the inline race/age dictionaries and duplicated region definitions embedded in `clean_dof_projections()` and `clean_census_estimates()`.

## Phase 2: Acquisition

### `scripts/projections/acquisition/dof_p3_downloader.py`

- **`P3DiscoveryError`** - raised when no P-3 zip link is found.
- **`get_p3_file_url(base_url, headers, timeout)`** - BeautifulSoup over the DoF page; returns the first `href` matching `P-?3.*\.zip`. *Derived from* `scrape_dof_projections()` (link-text matching).
- **`get_p3_file_url_positional(base_url, headers, timeout)`** - fallback that walks `div.et_pb_text_inner` containers, finds one mentioning "P-3", and returns its `.zip` link. *Derived from* `scrape_dof_projections_spatially()`.
- **`download_p3_data(url, download_directory, headers, timeout, cache_max_age_days)`** - returns a cached CSV inside the cache window, else downloads the zip, extracts the CSV, and deletes the zip. Uses `urllib.parse` + shared `download_file`.
- **`extract_csv_from_zip(zip_path, download_directory)`** - opens the archive with `zipfile.ZipFile`, requires **exactly one** CSV member (else `ValueError`), and streams it to disk. *This is new* - the legacy P-3 was an Excel workbook; the current distribution is a zipped CSV.
- **`get_most_recent_p3_file(download_directory, filename_pattern, max_age_days)`** - scans the directory for the newest CSV matching the pattern and within the age window, via `re.fullmatch` and `stat().st_mtime`.
- **`validate_p3_csv(csv_path, expected_columns)`** - reads only the header line, `Counter`s the columns, and raises if any mandatory column is missing or duplicated (checked before pandas can mangle duplicate names). Performance: header-only read, no full parse.

### `scripts/projections/acquisition/census_ccest_downloader.py`

- **`get_census_ccest_url(base_url, headers, timeout)`** - scrapes the Census datasets index for vintage subdirectories (`YYYY-YYYY/`), picks the **latest end year**, and constructs `.../{vintage}/counties/asrh/cc-est{end}-alldata.csv` (the nested listing is too large to scrape directly). *Derived from* `scrape_census_estimates()`.
- **`download_census_ccest(url, download_directory, headers, timeout, cache_max_age_days)`** - returns the cached file inside the cache window, else downloads via shared `download_file`.
- **`validate_ccest_headers(csv_path, expected_columns)`** - header-only membership check; raises on any missing identifier or population column.

### `scripts/projections/acquisition/source_fallback.py`

- **`acquire_with_fallback(live_strategies, manual_path, saved_rows_fn, source_name)`** - tries each live strategy (returns `(path, False, False)` on the first success), then a manual CSV (`(path, False, True)`), then last-saved rows (`(df, True, False)`). If even last-saved fails, raises a `RuntimeError` with every prior error attached via `add_note`. *Derived from* the legacy inline try/except fallback cascade inside `__init__`; generalized to a reusable coordinator matching the Components/PopHousing pattern.

## Phase 3: Cleaning

### `scripts/projections/cleaning/dof_p3_cleaner.py`

- **`map_fips_to_county(df, fips_column, fips_to_county_map, logger=None)`** - coerces FIPS to `Int64`, **drops and logs** any non-county FIPS (a state-summary or junk row a future vintage might add) rather than raising, then asserts every expected county FIPS is still present (a structural breach fails loudly), and replaces the column with `Location`. This is the value-level-drift-tolerant / structural-fail-closed posture applied across the cleaners.
- **`standardize_sex_labels(df, sex_column, label_map)`** - maps `MALE`/`FEMALE` to `Male`/`Female`; raises on unmapped values.
- **`bin_single_year_ages(df, age_column, population_column, bin_edges, groupby_columns)`** - `pd.cut` with `right=False` and an appended `inf` edge, then `groupby(...).sum()`. Labels are reconstructed from the edges by `_labels_from_bin_edges` (last edge becomes `"85+"`). This is the lossy aggregation.
- **`clean_p3_projections(csv_path, schema_config, logger=None)`** - the entry point: validate header, read **only** the mandatory columns via `usecols` (avoids loading unused columns and an extra copy), validate values (nulls, **observed** year range against soft sanity bounds + max span, age range, non-negative integer population), map FIPS/race/sex, bin ages **using the canonical labels directly**, tag `Geographic Level = County`, and return the canonical 7-column frame.
- Helpers `_validate_p3_header` and `_validate_p3_values` enforce the "each mandatory column exactly once, values semantically valid" contract.

*Legacy lineage:* replaces `clean_dof_projections()`. The legacy reshape step is gone because the current P-3 CSV is already one-row-per-record.

### `scripts/projections/cleaning/census_ccest_cleaner.py`

- **`parse_ccest_csv(csv_path, schema_config)`** - `pd.read_csv(..., encoding="latin-1")` (accented county names such as "Doña Ana" are not valid UTF-8), then validates required columns.
- **`aggregate_ccest_counties_to_states(df, schema_config)`** - filters to `SUMLEV=050`, keeps only the configured 50 states, confirms none are missing (else `ValueError`), and sums the 14 population columns by `(STNAME, YEAR, AGEGRP)`.
- **`rename_ccest_columns(df, schema_config)`** - renames `STNAME/YEAR/AGEGRP` to `Location/Year/Age Group` via the configured map.
- **`reshape_ccest_to_long(df, schema_config, logger=None)`** - builds the value-column list by parsing each `{RACE}_{SEX}` header (skipping `TOT`-prefixed totals; an unrecognized race prefix is **skipped and logged**, not fatal), excludes `YEAR=base` and `AGEGRP=0`, `melt`s to long, maps prefixes/suffixes to canonical race/sex, and **derives the calendar year arithmetically** (`base + code`), dropping-and-logging any code that decodes outside the sane range or any unknown AGEGRP code. The strict completeness gate downstream is the fail-closed backstop for a genuine loss of a whole state.
- **`clean_census_estimates(csv_path, schema_config)`** - chains the four steps above and tags `Geographic Level = US State`.

*Legacy lineage:* replaces `clean_census_estimates()`, formalizing the `SUMLEV=050` filter and the county-to-state aggregation the legacy code did inline.

### `scripts/projections/cleaning/race_ethnicity_mapping.py`

Shared by both cleaners. **`get_canonical_race_groups()`**, **`map_race_ethnicity(df, raw_column, source_code_map)`** (raises on unmapped codes), and **`validate_race_mapping_completeness(df, race_column)`** (returns `(is_valid, messages)`). Owns `CANONICAL_RACE_GROUPS` and `P3_RACE7_CODE_MAP`. *Derived from* the inline race-mapping dictionaries duplicated across the two legacy cleaners.

### `scripts/projections/cleaning/age_group_standardizer.py`

Owns `AGE_BIN_EDGES` and `CANONICAL_AGE_GROUPS`. **`get_canonical_age_groups()`**, **`get_age_bin_edges()`**, **`assign_age_group_from_single_year(df, age_column)`** (integer-division label assignment, `min(age // 5, top_bin)`; raises on ages outside 0-110), **`standardize_age_group_labels(df, raw_column, label_map)`**, and **`validate_age_group_completeness(df, age_column)`**. *Derived from* the inline age-reshaping logic in `clean_dof_projections()`.

> Note: the P-3 cleaner does its binning through `bin_single_year_ages`, which now takes the **canonical labels directly** (`labels=schema_config["canonical_age_groups"]`) rather than reconstructing them. The two label-construction routes are kept in sync by a module-import parity assertion in `schemas.py` (`_age_labels_from_edges(_AGE_BIN_EDGES) == _CANONICAL_AGE_GROUPS`), so any future edit that breaks parity fails at import / in CI.

## Phase 4: Merge and Aggregate

### `scripts/projections/merging/historical_merge.py`

- **`load_canonical_dataset(current_data_path)`** - (see Phase 1).
- **`load_historical_baseline(path)` / `combine_history_sources(baseline_df, current_df)`** - load the optional immutable seed and union it with the current output, preferring current rows on any overlap (contract-key dedup, `keep="last"`). Absent seed → empty frame → cold-start on current/live only.
- **`reduce_to_base_strata(df, schema_config)`** - single owner of "what is a base row": filters to `Geographic Level ∈ {County, US State}` and canonical age/sex/race. Used by both the fallback reducer and change detection so the two definitions can never diverge.
- **`combine_source_with_historical(new_df, historical_df, source, year_column, completeness_validator, schema_config)`** - stamps `Source`, runs the completeness validator on the incoming frame **before** touching history (raises on failure), then retains historical years absent from the incoming set — **reduced to base strata** so re-aggregation can never double-count retained Region/State/marginal rows — and appends the incoming years whole. No key-level upserts - replacement is atomic at `(Source, Year)` grain. *Derived from* the conditional-merge logic in `combine_dof_and_census()`.
- **`detect_new_source_data(new_df, historical_df, source, boundary_year, schema_config)`** - reduces **both** the incoming frame and history to base strata, then compares the set of contract-keyed rows beyond the boundary year. Because incoming is base-only and saved history is enriched, comparing without reducing would always differ; reducing both makes the flag truthful. Helper `_recent_row_set` builds the comparison set from `itertuples`.
- **`merge_dof_and_census(dof_df, census_df)`** - concatenates the two source frames, coerces `Year` to `int64`, and sorts. *Derived from* the final unify step of `combine_dof_and_census()`.

### `scripts/projections/aggregation/regional_aggregation.py`

- **`add_regional_data(df, regions_mapping, groupby_dimensions)`** - for each of the 9 regions, filters county rows, groups by the aggregation dimensions, sums `Population`, tags `Geographic Level = Region`, and appends.
- **`add_state_total(df, county_names, groupby_dimensions, state_name="California")`** - sums the 58 county rows into a `California` `State` row, but only for `(dimension)` keys that don't already have a DoF state row (so it never duplicates and never overwrites the separate Census `California`).

*Legacy lineage:* replaces the inline `add_region()` helper; the region membership now comes from shared geography rather than a copy-pasted dict.

### `scripts/projections/aggregation/precomputed_totals.py`

- **`add_all_ages_totals` / `add_both_sexes_totals` / `add_all_races_totals`** - each groups by "everything except the aggregated dimension and Population", sums, and stamps the aggregate label (`All Ages` / `Both Sexes` / `All`).
- **`build_precomputed_totals(df, schema_config)`** - runs the three in order (ages, then sexes, then races), each over the growing frame so the marginals nest correctly. Helper `_groupby_except` computes the group-by column list.

*This module is new* - the legacy code summed on the fly at chart time; storing marginals is a V3 performance decision to keep the browser from summing millions of rows.

## Phase 5: Finalize, Validate, and Save

### `scripts/projections/validation/projections_validators.py`

- **`validate_cleaning_output(df, schema_config)`** - a per-source post-clean validator (required columns, null **critical** columns, non-negative population, canonical race/age values). It is now **wired** into the fallback path (`_try_clean` gates every successful clean through it, so a structurally-wrong clean triggers the fallback cascade instead of surfacing later in the merge), and its null-count check now reads the `critical_columns` key the config actually defines (previously it read a non-existent `key_columns` and silently checked nothing).
- **`validate_projections_dataset(df, validation_config)`** - the final gate: required columns, row-count bounds, expected geographic levels, year range, and duplicate-key detection. Composes the shared validators.
- **`validate_stratification_completeness(df, schema_config)`** - the completeness gate used during merge: groups by `(Geographic Level, Location, Year, Source)`, counts distinct base `(Age, Sex, Race)` tuples against 252, and reports each incomplete group. Aggregate rows are excluded from the base count.

### `scripts/projections/output/finalize_dataset.py`

- **`assign_geographic_level(df, geography_config)`** - vectorized priority ladder (`Other` -> `Region` -> `County` -> `State` -> `US State`) via `Series.mask`, where the `US State` step is source-aware (`Source == "Census cc-est"` and location in the 50-state list). This is what lets California resolve to `State` for DoF and `US State` for Census.
- **`prepare_projections_output(df, schema_config)`** - checks contract columns, casts `Year` to `int64`, sorts, and returns columns in contract order.
- **`archive_and_save(df, current_path, archive_directory)`** - serializes the frame to CSV bytes, compares against the existing file by **streamed SHA-256** (no second full-file string in memory); on a match it returns `None` (no write, file left byte- and mtime-identical), otherwise it `shutil.copy2`s the old file into the archive with an `mm-dd-yy` stamp and writes the new bytes **atomically** (staged `.tmp` + `os.replace`), so a crash mid-write can never corrupt the contract file that doubles as history. The orchestrator only calls this when a source reported new data (the now-trustworthy change flags), so the full-file serialize + hash no longer runs on every idempotent invocation.

*Legacy lineage:* replaces the inline geographic-level tagging (legacy `np.select`-style ladder), the column ordering, and the conditional-save at the end of `combine_dof_and_census()`.

### `scripts/orchestrators/projections_pipeline.py`

- **`ProjectionsPipelinePhaseError` / `_raise_phase_error`** - tag failures with their phase.
- **Fallback and offline helpers:** `_load_saved_source`, `_try_clean` (runs a cleaner then gates its output through `validate_cleaning_output`, returning `None` to trigger the next fallback tier), `_clean_with_fallback` (uses the shared `reduce_to_base_strata` for the saved path), `_fallback_tier` (names the tier each source resolved to, for the run log), `_newest_matching`, `_dof_local_strategy` / `_census_local_strategy`, `_dof_live_strategies` / `_census_live_strategies` (the live strategies now header-validate each download via `validate_p3_csv` / `validate_ccest_headers` so a malformed file fails fast into the cascade), `_completeness_validator`, `_geography_config`.
- **`build_projections_dataset(config=None, logger=None)`** - runs the five phases, unions the seed + current output into history (Phase 1), threads the `logger` through cleaning and logs at the acquisition/cleaning/merge boundaries, derives `expected_levels` from which sources succeeded, gates `archive_and_save` on the trustworthy change flags, and returns `{dataset, dof_new_data, census_new_data, dof_failed, census_failed, output_path, row_count}`.
- **CLI:** `python -m scripts.orchestrators.projections_pipeline` prints the row count, stale-data warnings, and the write status.

*Legacy lineage:* replaces the entire `Projections.__init__()` ETL body, now split across pure functions with explicit phase boundaries and fallback flags.

## From Initial Refactor to Current State

The refactor first dissolved the legacy `Projections` class into the phase modules described above. A later reliability, robustness, and efficiency pass (2026-07-13, building on the 2026-07-10 audit) then hardened several parts of that initial refactored state into what this guide documents. This section records those changes: what the first-cut refactor left in place, what it became, and why. Two governance decisions set the posture throughout: **derive-and-accept** for source-format drift whose value is a stable arithmetic law (the Census year), and **degrade-gracefully for value-level surprises / fail-closed for structural breaches** (an unknown code or extra row is dropped-and-logged; a missing county, state, year, or an incomplete 252-cell matrix still fails the run). All of the below is implemented and covered by tests, and verified end-to-end against the live P-3 and cc-est files.

### Change detection now compares like grains, so the "new data" flags are meaningful

The first cut compared the **base-strata** cleaned rows (County / US State only) against the **fully enriched** saved history (which carries Region/State rollups and every `All Ages`/`Both Sexes`/`All`-race marginal). The incoming set was therefore always a strict subset of history, so `dof_new_data` / `census_new_data` were effectively always `True`; the pipeline stayed idempotent only because `archive_and_save` independently no-ops on a byte-identical SHA-256 — which meant a full re-serialization and hash of ~1.7M rows ran on *every* invocation, and the flags themselves were meaningless to any consumer that read them.

`detect_new_source_data` now reduces **both** sides to base strata (via the shared `reduce_to_base_strata`) before the set comparison, so an unchanged source correctly reports `False`. That single owner of "what is a base row" is shared with the fallback reducer, so the two definitions cannot drift. Because the flags are now trustworthy, the orchestrator gates the entire `archive_and_save` call on them: the expensive serialize + hash only runs when a source actually reported new data. The live offline run confirms it — a re-run against unchanged inputs logs `dof_new=False, census_new=False` and prints "No new data detected; file unchanged," with no serialize/hash performed.

### The Census year became decode-derived, and value-level surprises stopped discarding the whole file

`get_census_ccest_url` always builds the URL for the newest vintage on the Census index, but the decode was a capped table (`{code: 2018 + code for code in range(2, 8)}`, codes 2-7 → 2020-2025). The first time Census published `cc-est2026-alldata` (YEAR code 8), the cleaner would have raised `Unknown YEAR code(s): [8]`, dropped the whole national file to stale last-saved data, and silently frozen the series — even though the 2020-2025 rows were still valid.

Per the **derive-and-accept** decision, the year is now computed arithmetically (`census_year_base = 2018`, so calendar = `2018 + code`) and validated against a `census_year_sane_range` guard, so a new vintage ingests with no code change (code 8 → 2026) while a corrupt code decoding to, say, 2918 is dropped-and-logged rather than crashing the file. The same **degrade-gracefully** posture now covers the other value-level surprises the audit grouped together: an unrecognized race-column prefix is skipped-and-logged (the known races still form the complete matrix); an unknown AGEGRP code drops its rows; and in the P-3 cleaner a non-county FIPS (a state-summary or junk row a future vintage might add) is dropped-and-logged instead of raising. In every case the strict per-group 252-cell completeness gate is the **fail-closed** backstop: if the drop actually costs a whole county, state, or year, the run still fails loudly rather than shipping a short frame.

### The P-3 horizon tracks the observed vintage instead of a frozen 2020-2070

Both the cleaner and the final validator pinned the horizon to `(2020, 2070)`, so a future vintage with a shifted horizon (Baseline 2028 → 2025-2075, or an extension to 2075) would have failed cleaning validation and degraded to stale DoF data — the most damaging failure, since P-3 is the module's primary series. The cleaner now validates the **observed** `(min, max)` years against soft sanity bounds (`p3_year_sane_bounds = (2015, 2100)`) and a maximum span (`p3_max_year_span = 60`), so a plausible new horizon ingests automatically while a garbage year (9999) still fails. The final gate uses the same soft bounds, so the hard-coded 2070 no longer lives in two places.

### The three dead validators were wired, and one permanent no-op fixed

The audit found three defined-and-tested-but-unused validators. `validate_cleaning_output` was never called **and** read `config.get("key_columns", [])` while the config only defines `critical_columns`, so its null check was a permanent no-op. It is now wired into the fallback path — `_try_clean` runs each cleaner and gates its output through `validate_cleaning_output`, returning `None` (which triggers the next fallback tier) if the clean is structurally wrong — and it reads `critical_columns`, so the null check actually runs. The two download-time header guards, `validate_p3_csv` and `validate_ccest_headers`, are now called inside the live acquisition strategies, so a malformed download fails fast into the fallback cascade with a clear error instead of a deep cleaner traceback.

### A separate, seeded history path decoupled from the live output

`config/paths.py` set `historical_data_path == current_data_path`, so the pipeline read its own prior output as its baseline — a slow-corruption channel if a bad write ever became the next run's "truth." Deep history now lives in its own file, `DemographicProjections_Historical.csv` (`historical_data_path`, decoupled from `current_data_path`), read-only to the pipeline. Phase 1 loads both the current output and this seed and unions them with `combine_history_sources`, preferring the current output row-for-row while the seed supplies any deep year the current file may have lost. The seed is optional: on a fresh checkout it is absent and the pipeline **cold-starts** on live data plus whatever current output exists — a property this module already had (empty-contract bootstrap, no dropped years, strict completeness gate) and which the change deliberately preserves.

### The contract file is written atomically

`archive_and_save` overwrote the output in place with `write_bytes`, a regression against the sibling modules — and because the contract file doubles as history, a process killed partway through writing ~100 MB of CSV left a truncated file that became the next run's baseline. It now stages the new bytes to a sibling `.tmp` file and `os.replace`s it into place (atomic on the same filesystem), with the archive `copy2` still running first: archive old → write tmp → atomic replace. The streamed-hash no-op-on-match optimization is retained as a cheap final guard.

### Retained history is reduced to base strata before aggregation

`combine_source_with_historical` retained enriched historical rows for years absent from the incoming frame, and aggregation then ran over the full merged frame. `add_state_total` was guarded (it skips keys that already have a `State` row), but `add_regional_data` and the marginal builders were not, so a partial-year release would have double-counted Region and marginal rows. This was dormant only because both sources deliver their full year spans every run. The retention path now feeds aggregation the same base-strata shape a fresh clean produces (`reduce_to_base_strata` inside `combine_source_with_historical`), so Region/State/marginals are always rebuilt from base rows alone — closing the whole class of double-counting regardless of which builders were individually guarded.

### The two age-label routes can no longer drift

Age-group labels were built two ways — `_labels_from_bin_edges` on the hot path and `age_group_standardizer` in tests/helpers — with nothing enforcing that they agree. `bin_single_year_ages` now takes the canonical labels directly (`labels=schema_config["canonical_age_groups"]`), and `schemas.py` carries a module-import parity assertion (`_age_labels_from_edges(_AGE_BIN_EDGES) == _CANONICAL_AGE_GROUPS`) so any future edit that breaks parity fails at import and in CI.

### Structured logging gained full-phase coverage

The shared logging layer was already implemented and wired through `execute_pipeline_run` before this pass, but only the Phase 5 quality check emitted a record. The orchestrator now logs at the acquisition/cleaning/merge boundaries as well (`log_dataframe_info` / `log_message` / `log_processing_step`) and threads the logger into the cleaners, so the run record captures the ~1.7M-row output count, which fallback tier each source used, and any rows quarantined by the drop-and-log paths. A stale-fallback run is still recorded as `recovered` (the `dof_failed` / `census_failed` flags match `run_records`' `…_failed$` pattern).

## Current State

Everything described above is implemented and covered by tests: the full unit suite passes (1,066 tests) and `ruff` is clean on the touched modules. The whole pipeline was exercised end-to-end offline against the live P-3 and cc-est files (`PROJECTIONS_OFFLINE=1`): it cleans 745,416 DoF rows and 75,600 Census rows, aggregates and validates to **1,718,208 rows**, and — because change detection now compares like grains — logs `dof_new=False, census_new=False` against the unchanged history and skips the write entirely ("No new data detected; file unchanged"), so the costly full-file serialize + hash no longer runs on an idempotent invocation.

The eleven audit items are closed. Reliability: change detection compares base-vs-base grains (A1) and gates the serialize/hash on the result (B3); the contract file is written atomically (B1); the Census year is decode-derived with a sane-range guard (A2); the P-3 horizon tracks the observed vintage within soft bounds (A3); the FIPS map drops-and-logs non-county rows while asserting all 58 counties (A7); a consistent "value-level surprise → drop-and-log, structural breach → fail" policy spans the cleaners (B2); and retained history is reduced to base strata before aggregation (A8). Housekeeping: the three dead validators are wired and the `key_columns` no-op fixed (A4); deep history has its own immutable seed decoupled from the live output (A5); the canonical age labels are a single source of truth with an import-time parity assertion (B4); and the cold-start + completeness-gate strengths are preserved and regression-tested (B5).

What remains is operational rather than structural: the immutable history seed (`DemographicProjections_Historical.csv`) is committed as a frozen copy of a known-good current output and refreshed only when it needs to absorb newer deep history; on a fresh checkout without the seed, the pipeline proceeds on live plus current data. When Census first publishes a `cc-est2026-alldata` vintage, the new YEAR code will now ingest automatically rather than freezing the national series.
