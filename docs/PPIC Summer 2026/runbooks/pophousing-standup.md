# Standing Up Population & Housing From Scratch

This runbook covers the one path the pipeline could not previously document: bringing Population & Housing up on a machine where no data yet exists. It resolves the coupled cold-start gaps flagged as A1, A2, and B8 in [[pophousing-pipeline-refractor]].

> [!info] Who this is for
> An operator or developer setting up the pipeline on a clean checkout, or recovering a deployment whose data went missing. It assumes the Python environment is installed (`.venv` active) and commands run from the repository root.

---

## The Two Datasets

The pipeline reads and writes two separate files, and confusing them is the root of the cold-start problem.

| File | Written by | Role |
|---|---|---|
| `PopHousing_Historical_E8.csv` | The Phase 0 builder only | Immutable pre-2020 history. The main pipeline reads it and never writes it. |
| `PopHousing_Current.csv` | The main pipeline only | The published dataset (1991-present). Overwritten every run. |

Both live under `data/data-cleaned/housing-population/`. The historical baseline has a sidecar, `PopHousing_Historical_E8.meta.json`, recording its coverage and how it was built.

> [!important] The baseline is a prerequisite, not a pipeline output
> The main pipeline cannot bootstrap its own history. If `PopHousing_Historical_E8.csv` is absent, Phase 1 aborts with an actionable error pointing back to this runbook. Build the baseline first.

---

## Step 1: Seed the Baseline

There are two ways to produce the baseline. On an existing deployment that already has a good `PopHousing_Current.csv`, seed from it - this reproduces the exact pre-2020 history the pipeline used before the baseline was split out.

```bash
python -m scripts.pophousing.historical.build_baseline --from-current
```

On a truly clean machine with no prior output, rebuild from the DoF E-8 decade workbooks instead. Place the workbooks named in `HISTORICAL_FILE_CONFIG` under `data/data-raw/housing-population/`, or pass `--download` to fetch them:

```bash
python -m scripts.pophousing.historical.build_baseline            # workbooks already local
python -m scripts.pophousing.historical.build_baseline --download  # fetch from DoF first
```

Either mode validates the result against the shared historical-validation gate before writing, writes the baseline atomically, and refreshes the sidecar. A failed validation refuses to write, so a bad build never lands.

---

## Step 2: Verify the Gate

The baseline is only useful if it passes the same acceptance check the main pipeline's Phase 1 runs. The builder already applies it, but you can confirm independently by reviewing the printed summary (row count and year range) and, for a rebuild, diffing the new baseline against the previous committed version before committing. The gate checks year coverage back to 1991, the presence of State, County, and City levels, a plausible recent California population, zero nulls in required columns, and duplicate keys.

> [!tip] Review a rebuild as a data diff
> When rebuilding from workbooks, treat the new `PopHousing_Historical_E8.csv` as a reviewable artifact. Compare it against the committed version, confirm the row counts and per-level counts are sensible, then commit it deliberately. The baseline is versioned precisely so this review is possible.

---

## Step 3: Run the Main Pipeline

With the baseline in place, the normal pipeline runs end to end:

```bash
python -m scripts.orchestrators.pophousing_pipeline
```

Phase 1 confirms the baseline exists, logs a freshness check, and validates it; the remaining phases acquire the current E-5 workbook, clean it, merge it onto the baseline, enrich, and write `PopHousing_Current.csv`.

---

## When to Rebuild the Baseline

The baseline is deliberately static. Rebuild it only when the historical source changes - which, for DoF E-8, is roughly once a decade plus corrections.

> [!note] The freshness warning tells you when
> Every run compares the baseline's newest year against the newest `year_end` in `HISTORICAL_FILE_CONFIG`. When a new E-8 era is wired into the config but the baseline still predates it, Phase 1 logs a warning. That warning - not a calendar - is the signal to rerun Step 1's rebuild path. A future frontend "update data" control will trigger the same rebuild without touching the codebase.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `Phase 1 failed: Historical E-8 baseline not found` | Cold start; baseline never seeded. | Run Step 1. |
| `Missing E-8 workbook(s) required to rebuild` | Rebuild mode cannot find the decade workbooks. | Place them under `data/data-raw/housing-population/` with the `HISTORICAL_FILE_CONFIG` filenames, or pass `--download`. |
| `Historical baseline failed validation; refusing to write` | The seed or rebuild produced data that fails the gate. | Inspect the printed messages; fix the source, do not force-write. |
| Freshness warning on every run | `HISTORICAL_FILE_CONFIG` extends past the committed baseline. | Rebuild the baseline (Step 1, rebuild path) and commit it. |
